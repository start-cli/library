# Create README.md

Create a README.md for the current working directory. A human-focused document that helps people understand, install, and use the project. The directory may be a repository, a package subdirectory, a standalone tool, or any other project structure.

## Instructions

1. Confirm no README.md file exists in the current working directory
2. Follow the process below to create one

## Process

Create a new README.md by:

1. Explore the directory to understand the technology stack and project type
2. Read any existing documentation (docs/, CONTRIBUTING.md, LICENSE, etc.)
3. Identify the project type from the list below
4. Select appropriate sections for that project type
5. Write the README.md

## Project Types and Sections

Identify the project type and apply the matching section set. Omit sections that are not relevant.

### CLI Tool

Sections: title and description, badges, installation, usage (with command examples), configuration, contributing, license

### Library or Package

Sections: title and description, badges, features, installation, quick start, API reference or usage examples, contributing, license

### Web Application

Sections: title and description, badges, features, screenshots (placeholder links), prerequisites, installation and setup, configuration, deployment, contributing, license

### Service or API

Sections: title and description, badges, features, prerequisites, installation and setup, configuration, API overview, deployment, contributing, license

### Internal or Private Repository

Sections: title and description, overview, prerequisites, setup and installation, development workflow, architecture overview, team and contacts

### Generic or Mixed

Use: title and description, features, installation, usage, contributing, license

## Style Guidelines

README.md is a human-facing document. Use full CommonMark as appropriate:

- Use a descriptive project title as the main heading, not "README" or "README.md"
- Keep the description concise - one to three sentences that explain what the project does and who it is for
- Use badges sparingly and only where they add value (build status, version, license)
- Prefer real command examples over abstract descriptions
- Use code blocks with language identifiers for all code and commands
- Include a license section that names the licence and links to the LICENSE file
- Avoid filler sections with placeholder text - only include a section if you have content for it
- Keep contributing and license sections brief; link to separate files where they exist
